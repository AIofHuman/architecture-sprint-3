# Задание 1. Анализ и планирование

<aside>
💡

«Тёплый дом» — это небольшая компания, которая организует удалённое управление отоплением в доме.

Нынешнее приложение компании позволяет только управлять отоплением в доме и проверять температуру.
Каждая установка сопровождается выездом специалиста по подключению системы отопления в доме к текущей версии системы.
Архитектура приложения представляет из себя монолит на Java с СУБД Postgres. Всё синхронно. Никаких асинхронных вызовов, микросервисов и реактивного взаимодействия в системе нет. Всё управление идёт от сервера к датчику. Данные о температуре также получаются через запрос от сервера к датчику.
Самостоятельно подключить свой датчик к системе пользователь не может.

</aside>

### 1. Описание функциональности монолитного приложения



Пользователи могут:
* Пользователи могут удалённо включать/выключать отопление в своих домах
* просматривать текущую температуру в своих домах через веб-интерфейс
<br>
Система поддерживает:
* удаленные команды включения/выключения нагревателей
* получения информации о текущей температуре в доме


### 2. Анализ архитектуры монолитного приложения

- Язык программирования: Java
- База данных: PostgreSQL
- Архитектура: Монолитная, все компоненты системы (обработка запросов, бизнес-логика, работа с данными) находятся в рамках одного приложения.
- Взаимодействие: Синхронное, запросы обрабатываются последовательно.
- Масштабируемость: Ограничена, так как монолит сложно масштабировать по частям.
- Развёртывание: Требует остановки всего приложения.

### 3. Определение доменов и границы контекстов

Есть два домена:
- Управление отопительным оборудованием
- Мониторинг текущей температуры

### **4. Проблемы монолитного решения**

### Основные проблемы текущей монолитной архитектуры

1. **Сложность масштабирования**
* Невозможно масштабировать отдельные компоненты системы независимо друг от друга
* При росте нагрузки приходится увеличивать мощности всего сервера, даже если проблема только в одном модуле
* Ограниченная возможность горизонтального масштабирования

2. **Проблемы с развертыванием**
* Любое обновление требует остановки всей системы
* Высокий риск нарушения работы всех функций при обновлении даже небольшой части кода
* Длительное время простоя системы при деплое

3. **Зависимость компонентов**
* Изменения в одной части системы могут повлиять на работу других модулей
* Сложно изолировать ошибки и баги
* Замедление разработки из-за необходимости тестирования всей системы после каждого изменения

4. **Ограничения производительности**
* Синхронное взаимодействие создает узкие места при обработке запросов
* Очереди запросов могут приводить к задержкам в обработке команд отопления
* Отсутствие асинхронности ограничивает возможности оптимизации производительности

5. **Сложности с интеграцией новых функций**
* Добавление новых типов датчиков требует полной модификации системы
* Невозможно быстро внедрить новые возможности без остановки всего сервиса
* Высокая стоимость поддержки и развития системы

6. **Операционные проблемы**
* Необходимость выезда специалиста для каждой установки
* Отсутствие самообслуживания для пользователей
* Высокие операционные расходы на поддержку

7. **Риски безопасности**
* Одна уязвимость может затронуть всю систему
* Сложно изолировать критические компоненты
* Ограниченные возможности по разделению доступа и защите данных

8. **Проблемы с тестированием**
* Сложность написания изолированных тестов
* Необходимость тестирования всей системы при каждом изменении
* Затраты времени на развертывание тестовой среды

9. **Ограничения по технологиям**
* Привязка к одному стеку технологий (Java + Postgres)
* Сложность внедрения новых технологий и подходов
* Ограничения в выборе инструментов разработки

10. **Проблемы с отказоустойчивостью**
* Отказ одного компонента может привести к сбою всей системы
* Отсутствие механизмов автоматического восстановления
* Сложность организации резервного копирования и восстановления
